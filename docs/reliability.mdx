title: "Reliability and production readiness"
description: "Understand Vell's current guarantees, known limitations, compatibility policy, and the work required before a stable 1.0 release."
---

Vell has a working language toolchain today. You can parse, validate, format, render, and edit `.vl` documents with the repository's Rust and TypeScript components. The project is still version `0.1.0`, so production adoption should use pinned versions and regression tests.

## What is reliable today

| Area | Current guarantee |
|---|---|
| Parser | Deterministic AST output for identical UTF-8 source and parser version |
| Source spans | Byte offsets into the original UTF-8 source |
| Formatter | Idempotent output across the fixture and specification suites |
| Unknown extensions | Preserved as `Extension` nodes instead of rejected |
| HTML safety | User text is escaped and unsafe URL schemes are rejected |
| CLI errors | Parse and I/O failures return a non-zero exit code |
| Includes | Recursive includes detect circular paths |
| LSP | Diagnostics, formatting, navigation, rename, and ranged document edits |
| Unicode | Multilingual text survives parse and format operations without byte corruption |

## Output maturity

<ResponseField name="HTML" type="Stable core">
  The broadest renderer. It supports MathML, references, footnotes, navigation lists, semantic book sections, index and glossary generation, diagrams, charts, and optional interactivity.
</ResponseField>

<ResponseField name="PDF package" type="Usable with limitations">
  `@solez-ai/vell-renderer-pdf` creates PDF bytes directly with PDFKit. Complex mathematical typography, diagram engines, font embedding, and advanced book layout still need more work.
</ResponseField>

<ResponseField name="CLI PDF" type="Print-HTML workflow">
  `vell render pdf` currently writes PDF-optimized HTML. Convert that HTML with your browser's print dialog or a trusted HTML-to-PDF tool. It does not emit PDF bytes.
</ResponseField>

<ResponseField name="EPUB" type="Usable baseline">
  Produces EPUB 3 archives with chapter splitting and navigation. Validate important releases with EPUBCheck before distribution.
</ResponseField>

<ResponseField name="DOCX" type="Baseline exporter">
  Produces a valid Office Open XML archive with text, headings, lists, code, and math source. It does not yet preserve every Vell style, image, table, reference, or complex layout feature.
</ResponseField>

<ResponseField name="LaTeX" type="Baseline exporter">
  Produces a standalone `.tex` file. Inline formatting fidelity, package selection, bibliography workflows, and complex directive conversion require additional coverage.
</ResponseField>

<ResponseField name="JATS" type="Baseline exporter">
  Produces JATS 1.3-oriented XML. Journal submission still requires validation against the journal's selected JATS DTD and house rules.
</ResponseField>

## JavaScript packaging status

The `@solez-ai/vell` package contains the typed wrapper. It does **not** currently contain a generated `.wasm` binary or an `@solez-ai/vell/wasm` export. Build `vell-wasm` with `wasm-pack`, import the generated module, then pass it to `setWasmModule()`.

This is the highest-priority packaging gap before the JavaScript API can be considered install-and-use.

## Interactive security boundary

- Computed variables compile document-authored expressions with `Function` in the page context.
- `@[On]` uses a constrained assignment, increment, decrement, and toggle action grammar.
- `@[Run]` and executable code fences run JavaScript inside a sandboxed iframe.
- Interactive HTML should only be enabled for documents you trust or have reviewed.
- Static HTML rendering does not execute document code.

<Warning>
  Do not enable interactive rendering for arbitrary untrusted documents. HTML escaping prevents markup injection, but intentionally executable document features are still code.
</Warning>

## Compatibility policy before 1.0

Until Vell reaches `1.0.0`:

1. Pin Rust crates and npm packages to an exact version.
2. Store representative `.vl` fixtures in your project.
3. Snapshot the AST and important rendered outputs.
4. Run `vell validate` and `vell fmt --check` in CI.
5. Review release notes before upgrading.

Breaking AST changes must increment the AST schema version. Renderers should continue to tolerate unknown nodes by preserving children or emitting a neutral fallback.

## Work required for a dependable 1.0

The following work has the highest impact on real users:

1. **Ship the WASM binary with `@solez-ai/vell`** and test Node.js, browsers, Vite, webpack, and Deno.
2. **Add real TypeScript behavioral tests.** Current npm tests primarily type-check packages.
3. **Add fuzzing and property tests** for parser panic resistance, formatter idempotency, and hostile inputs.
4. **Define a formal compatibility suite** for AST JSON, diagnostics, formatting, and renderer output.
5. **Improve publishing fidelity** for DOCX, JATS, LaTeX, EPUB, and direct PDF output.
6. **Add release automation** for crates, npm packages, WASM artifacts, VS Code packages, checksums, and changelogs.
7. **Publish supported-platform matrices** with minimum Rust, Node.js, browser, and editor versions.
8. **Add performance benchmarks and limits** for deeply nested, very large, and adversarial documents.
9. **Complete accessibility testing** with keyboard navigation, screen readers, CJK fonts, RTL documents, and high-contrast modes.
10. **Add migration and deprecation rules** before declaring the language and AST stable.

## Recommended adoption level

| Use case | Recommendation |
|---|---|
| Personal documents and experiments | Recommended |
| Internal documentation with pinned versions | Recommended with CI snapshots |
| Public HTML documentation | Usable after security and browser testing |
| Automated academic publishing | Pilot only; inspect PDF and bibliography output |
| Journal submission or regulated documents | Not yet without downstream validation |
| Untrusted interactive document hosting | Not recommended |

Vell is a real working language implementation. Reaching a dependable 1.0 now depends more on packaging, compatibility, testing depth, and output fidelity than on adding more syntax.
